From 15 hours to 30 minutes:
Building a 6-agent AI training pipeline

How I designed and shipped an autonomous content-production system that turns a single training topic into a quality-audited, branded Canva deck — with three human checkpoints and zero deadline misses.

RoleSole architect & builder
Team1 (me)
StackPython, FastAPI, Claude, Canva API
StatusIn production since Q4 2025

TL;DR

Producing a single bi-monthly VA training deck used to take 15+ hours of research, writing, simplifying, slide copy, design briefing, and quality review. I built a 6-agent pipeline on FastAPI where each agent owns one production stage. End-to-end runs now finish in 20–47 minutes. Three human-in-the-loop checkpoints preserve quality. Zero missed deadlines since deployment.

The problem

I run development & experience for 70+ virtual assistants serving Australian businesses. Every two weeks, the team needs a polished training deck on a new topic — mindset, AI tooling, or workflow upgrades. The old workflow was painfully manual:

  1. Research the topic across reputable sources (~2–3 hours)
  2. Write the trainer script in our voice (~3–4 hours)
  3. Simplify for VA reading level & ESL friendliness (~1–2 hours)
  4. Slide-ify the script into deck-friendly copy (~2–3 hours)
  5. Brief a designer / build the Canva deck (~3–4 hours)
  6. Quality-review against rubric (~1–2 hours)

Total: 12–18 hours per deck. Repeat every 14 days. With everything else on my plate, deadlines slipped, quality varied, and I burned a Saturday on it more than once.

The architecture

I split the work into 6 specialized agents, one per production stage. Each agent has a single responsibility and a structured output the next agent can consume:

SCOUT → FORGE → CLEAR → QUILL → PIXEL → LENS

SCOUT

Researches the topic, classifies it (Mindset / AI Skillset / Workflow), and emits a structured topic brief.

FORGE

Writes the full trainer script in one of three voice modes matched to the topic class.

CLEAR

Simplifies the script for VA reading level — shortens sentences, swaps jargon, preserves meaning.

QUILL

Polishes the simplified script into slide-ready copy with titles, bullets, and speaker notes.

PIXEL

Builds a structured Canva design brief: layouts, imagery directions, brand color use, slide order.

LENS

Audits the final deck against a 50-point quality rubric. Pass threshold: 38/50. Fails route back.

Human-in-the-loop checkpoints

I deliberately added three approval gates — not because the AI is unreliable, but because content for live trainers should never run unattended:

  • After SCOUT: approve the topic angle & classification before script generation
  • After QUILL: approve slide copy before design brief
  • After LENS: approve the audit and any flagged revisions before publish

Implementation notes

FastAPI as the orchestrator

Each agent is a FastAPI endpoint that takes the previous stage's output as JSON, calls Claude through OpenRouter, and returns structured output. The pipeline state lives in a simple JSON file per run — no Postgres, no Redis. For a single-user system processing one deck at a time, that's enough.

POST /agents/scout      → { topic, class, brief, sources }
POST /agents/forge      → { script, mode, voice_notes }
POST /agents/clear      → { simplified, reading_level, removed_jargon }
POST /agents/quill      → { slides[], speaker_notes[] }
POST /agents/pixel      → { design_brief, layouts, color_map }
POST /agents/lens       → { score, breakdown, flags[] }

Model selection per stage

Not every stage needs the most expensive model. SCOUT and CLEAR run on Sonnet (cheaper, faster, good enough). FORGE and LENS run on Opus (the writing & auditing stages where quality compounds). PIXEL also runs on Sonnet. This cuts cost ~60% versus running everything on Opus.

The watchdog

Windows Task Scheduler keeps FastAPI alive: a 1-minute polling task checks /health and restarts the uvicorn process on failure. Crude, but it has a 99%+ uptime over months. No need for Docker or a process manager when one user is running it locally.

Design choice: I picked simplicity over scalability everywhere I could. JSON files instead of a database. Local FastAPI instead of cloud. Task Scheduler instead of systemd or PM2. The system serves one user (me) producing one deck at a time. Optimizing for scale would have cost weeks; optimizing for shipping cost days.

Missed deadlines

Beyond raw time savings, the second-order effects mattered more:

  • Consistency: the rubric & voice modes mean every deck hits the same quality bar
  • Reusability: SCOUT outputs feed the SEO content engine; CLEAR outputs feed our help-doc rewrite project
  • Reclaim: ~25 hours/month back, redirected to coaching VAs and shipping more systems like this one

What I'd do differently

Build the audit agent (LENS) first

I built it last. Big mistake. Having a working rubric earlier would have made every other agent's prompt-tuning faster — I would have had a measurable target instead of vibes.

Structured outputs from day one

Early agents returned free-form text that the next agent had to re-parse. Forcing JSON schema outputs cut errors by ~70%.

One agent per personality, not per task

FORGE has three voice modes inside it (Mindset / Skillset / Workflow). I almost split into three agents. Keeping them in one with a mode flag was the right call — less duplication, easier prompt iteration.

The watchdog is non-negotiable

The first version had no auto-restart. The first time uvicorn died at 11pm before a 9am deadline, I added it the next morning.

Stack used

  • Python + FastAPI — orchestration layer
  • Claude (Opus 4.6, Sonnet 4.6) — agent intelligence via OpenRouter
  • Canva API — deck assembly from PIXEL's brief
  • Windows Task Scheduler — watchdog & cron
  • JSON files — pipeline state (no DB)
